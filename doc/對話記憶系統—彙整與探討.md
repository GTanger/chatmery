## 共識與最適合本專案的作法（收斂摘要）

以下收斂自：對話記憶與背版—設計，以及本文件 §三～§七。

### 共識

| 面向 | 共識內容 |
|------|----------|
| **記憶主體** | 每對話主體一 **subject**（`subject_id`）；背版 = **blocks**（identity／summary／relationship）；長期記憶 = **archival**，語意檢索後取 top-k 注入 prompt。（設計 doc、本文件 §7） |
| **背版與口吻** | 若有**口吻／例句**，負責語氣與風格；**背版＋記憶**負責「是誰、發生過什麼」；組 prompt 時背版＋精選記憶＋口吻並存。 |
| **寫入時機** | 可每輪寫 0～1 條，或**對話結束時 consolidation** 寫 1～3 條；不逐句寫、節流，避免轟炸造成 archival 噪音。（本文件 §7.4） |

### 最適合本專案的作法（建議路徑）

| 階段 | 作法 | 說明 |
|------|------|------|
| **短期（先上線）** | 固定背版（identity）**只讀**；回覆時背版＋最近 1～2 輪組 prompt；可選：本輪或結束時寫 1～3 條進 archival。 | 回覆有人設、延續性；背版尚不隨對話成長；小模型 context 有限，最近輪數建議 1～2。（設計 doc 階段 1～2） |
| **中期** | archival **語意檢索**（embedding＋向量庫），回覆前用使用者輸入或話題當 query 取 top-k（如 3～5 條）注入；**對話結束觸發 consolidation**，將本場精華寫入該主體的 archival；可選更新 **relationship** block。 | 越聊越立體；檢索線索 = 編碼特定性；節流與 consolidation 見 §7.4。（設計 doc 階段 2） |
| **長期** | 定時或每 N 次對話，由 **LLM 或規則**整理 archival → 寫回 **summary／relationship** blocks；背版總長度控制（每塊數十字～百字），避免爆 context。 | 要旨式背版、token 可控；對應人類記憶的「鞏固與摘要」。（設計 doc 階段 3、本文件 §1.6） |

**實作接點**：組 prompt 時傳入 `backstory`、`memorySnippets`；對話流程內「讀背版＋檢索記憶→組 prompt→回覆→可選寫入／consolidation」；記憶寫入／檢索 API 依 subject_id 分區。依賴 embedding（可本地如 bge-m3）與向量庫。

## 一、人類記憶系統運作流程（認知心理學）

以下為認知心理學中人類記憶的經典模型與流程，供設計「對話／Agent 記憶」時對照與類比。

### 1.1 三階段流程：編碼、儲存、提取

| 階段 | 英文 | 說明 |
|------|------|------|
| **編碼** | Encoding | 將感官與經驗轉成記憶系統可處理的形式。常見編碼類型：**視覺**（圖像）、**聽覺**（聲音）、**語意**（意義）。短期記憶偏聽覺編碼，長期記憶偏**語意編碼**。 |
| **儲存** | Storage | 在時間中維持資訊，依系統不同有不同容量與保存期。 |
| **提取** | Retrieval | 在需要時取回已儲存的資訊。提取成功與否受編碼方式與**提取線索**影響。 |

### 1.2 Atkinson–Shiffrin 三儲存模型（1968）

資訊依序流經三種儲存，並受**控制歷程**（注意力、複誦、編碼、提取）調節：

| 階段 | 名稱 | 容量 | 持續時間 | 說明 |
|------|------|------|----------|------|
| **感覺登錄** | Sensory Register | 極大 | 毫秒～數秒 | 接收所有感官輸入；只有被**注意**的資訊才進入下一階段，其餘迅速衰退。 |
| **短期儲存** | Short-Term Store (STS) | 有限（約 5～9 個單位） | 約 0～30 秒 | 又稱工作記憶；可經**複誦**維持，但單純複誦未必賦予意義。可從長期儲存讀回以支援當下任務。 |
| **長期儲存** | Long-Term Store (LTS) | 視為無限 | 可終身 | 經複誦與**意義化**的資訊在此持久保存。 |

**要點**：資訊須依序通過此流程才能形成持久記憶；控制歷程決定「什麼被注意、什麼被複誦、什麼被編碼與提取」。

### 1.3 長期記憶的類型

| 類型 | 說明 | 範例 |
|------|------|------|
| **陳述性記憶**（外顯） | 可意識回想的事實與事件 | **情節記憶**：與特定時間、地點綁定（「上週二在店裡聊過」）；**語意記憶**：一般知識（「這條街有很多商鋪」）。 |
| **非陳述性記憶**（內隱） | 不易言傳的習慣與反應 | 程序性記憶（技能）、情緒制約等。 |

對話與背版設計較常對應**陳述性**：情節（某次對話、某件事）+ 語意（對某人／某地的概括認識）。

### 1.4 記憶鞏固（Consolidation）與睡眠

- **系統鞏固**：新記憶先依賴海馬迴，隨時間逐漸**轉存到皮質**，成為較穩定的長期記憶。
- **睡眠角色**：睡眠中（尤其慢波睡眠）會**重播**日間經驗、強化重要記憶，並與既有知識整合；同時發生**選擇性遺忘**（不重要的細節被弱化）。
- **質變**：鞏固後記憶常從「逐字逐句的情節」變成**要旨式、摘要式**表徵，細節減少但結構保留。

### 1.5 提取線索與情境依賴

- **編碼特定性**：提取時若**線索或情境**與編碼時重疊，回憶較佳。例如在同一地點、同一情緒狀態下較易想起當時的事。
- **提取線索**：可以是外界提供（問題、關鍵字、場景）或自我產生；線索與**編碼時綁定的資訊**重疊越多，越容易喚起。
- **情境依賴記憶**：編碼時的情境（地點、周遭、身心狀態）若在提取時再現，有助回憶。

### 1.6 與對話記憶設計的對照

| 人類記憶 | 對話／Agent 記憶設計可類比 |
|----------|----------------------------|
| 感覺登錄 → 注意篩選 | 每輪對話的原始訊息；只將「值得記的」寫入（篩選／LLM 抽取）。 |
| 短期／工作記憶 | **當前 context 視窗**：最近幾輪對話、當下背版摘要；容量與 token 上限對應。 |
| 長期儲存 | **Archival**（向量庫）＋可選 **blocks**（摘要區塊）；容量可視為大或無限。 |
| 編碼（尤語意） | 寫入時用 **embedding** 或 **tag** 做語意／分類編碼，方便日後依「意思」檢索。 |
| 提取與線索 | **語意搜尋**：用當前話題或玩家輸入當 query，拉出與編碼重疊的記憶（top-k）。 |
| 鞏固與摘要 | **定時或觸發**：將最近對話或 archival 精華交給 LLM 產出摘要，寫回 **blocks**（要旨式、省 token）。 |
| 情境／線索重疊 | 檢索時可帶入**當前情境**（房間、玩家 id、話題）當 query 或 filter，提高相關記憶被取出的機率。 |

## 二、AI 如何解析與解構人類語言（以中文為例）

本節說明：當你對 AI 說出一段話時，模型在技術上如何「讀取」與表示這段話，以及從**輸出端**可觀察到的「分析／解構」是什麼；並說明這與**對話記憶**的關聯（編碼什麼、用什麼線索提取）。

### 2.1 讀取：從字串到 token 再到向量

- **Tokenization（分詞／切符）**：輸入的中文與標點會先被切成一串 **token**。中文沒有空格斷詞，常見做法是 **BPE 或類似子詞演算法**：一個中文字可能對應一個或多個 token，詞組（如「重體驗輕操作」「代碼生成」）可能被切成多個子詞或整詞，依訓練時詞表與合併規則而定。例如「我在使用AI代理」可能變成 `我` `在` `使用` `AI` `代理` 或更細的切分。
- **向量表示**：每個 token 對應一個**向量**（embedding）；整段序列送入 **Transformer** 後，透過自注意力（self-attention）讓每個位置都參照到整段上下文，產出**帶上下文的表示**。也就是說：模型不會先跑一套「文法剖析樹」或「語意角色標註」的管線，而是用**分佈式表示**把句法、語意、主題、焦點等編進高維向量裡；「理解」是隱含在這些表示與後續生成行為中，而不是顯式的結構化解構。

### 2.2 從輸出端看「分析／解構」：以你的例子為例

你寫的這段話：

> 我在使用AI代理製作重體驗輕操作的MUD遊戲，但是因為我不懂代碼，我只能反覆與AI討論架構並生成文檔，然後AI再依憑文檔作為代碼生成的指令，但也因為我看不懂代碼，我無法中途查錯，只能依靠成果驗收。

從**模型如何回應**這件事往回推，可以說模型至少「用到了」以下幾類資訊（這些相當於一種**解構**）：

| 層面 | 內容（解構出的要點） |
|------|----------------------|
| **主題／專案** | 在做的是「MUD 遊戲」、取向是「重體驗輕操作」、手段是「AI 代理」。 |
| **工作流程** | 反覆與 AI 討論架構 → 產出文檔 → 文檔當作「給 AI 的程式碼生成指令」→ 無法讀碼、無法中途除錯 → 只能靠**成果驗收**。 |
| **約束／用戶狀態** | 不懂代碼、看不懂代碼；因此無法做程式碼層級的查錯，只能依賴文檔與最終結果。 |
| **因果／動機** | 「因為……所以只能……」：能力限制（不懂碼）導致策略（文檔驅動、成果驗收）。 |

模型並沒有先輸出「主詞＝我、動詞＝使用、受詞＝…」這種語法樹，而是**在生成時**表現出能抓住主題、流程、約束與因果；這來自訓練中學到的對類似句子的表示與對應行為。

### 2.3 與對話記憶的關係

- **要記什麼（編碼）**：若這段話要寫進長期記憶，我們不會存「每個 token 的向量」，而是存**語意層的摘要**，例如：「用戶在用 AI 代理做重體驗輕操作的 MUD；不懂代碼，流程是討論架構→生成文檔→文檔驅動程式碼生成→只能成果驗收。」這樣日後用「MUD」「文檔驅動」「不懂代碼」等**語意相近的 query** 才能把這段記憶拉出來——對應人類記憶的**語意編碼**與**提取線索**。
- **用什麼線索提取**：若之後你問「我上次說我怎麼做遊戲的？」或「我不會寫程式怎麼辦？」，檢索時會用當前句做 **embedding** 或關鍵詞，在向量庫裡找**語意相近**的已存段落；也就是依「意思」而非逐字匹配，和人類的**編碼特定性**類似。
- **為何放在同一份文檔**：對話記憶的設計（寫入什麼、用什麼檢索、如何摘要）直接依賴「模型如何表示與使用語言」：我們存的是**語意層的壓縮**，檢索靠**語意相似度**，和模型內部的 token／向量表示同一套「語意空間」；因此把「AI 如何解析／解構語言」放在對話記憶系統文檔中，有助於一致地設計編碼與提取策略。若日後要擴充為「多語、多模態、或顯式語意角色」，再考慮獨立成「語言理解與表示」專檔即可。

## 三、AI Web Chat 與 Cursor 的對話／記憶機制（對照）

**對照前提**：Gemini web chat、Claude web chat、Grok、ChatGPT、以及 Cursor 的**當前對話視窗**，在本文件類比下都相當於**短期記憶池**——即單次 session 內可見的對話與被注入的 context，有 token 上限，滿了會截斷或壓縮舊訊息。差異在**長期記憶池**的來源與注入方式。以下彙整各產品機制並對應「短期／長期」與流程。

### 3.1 共通：當前對話視窗 = 短期記憶池

| 產品 | 短期記憶池（當下 session） | 說明 |
|------|----------------------------|------|
| **ChatGPT** | 單一對話串內的訊息序列；有 context 上限（如 128K～400K token，依模型） | 超出時舊訊息被截斷或壓縮；可開「臨時對話」不寫入長期。 |
| **Claude（web）** | 同上；context 視窗依模型（如 200K） | 對話歷史在 session 內保留，滿了從頭丟棄或摘要。 |
| **Gemini（web）** | 同上；並有「Past Chats」可參考過往對話 | 個人化會參考過去聊天，但**當前視窗**仍是本串訊息。 |
| **Grok** | Session 內訊息；context 約 128K～131K token | 關閉 session 後 session 內內容清除。 |
| **Cursor** | 當前 Chat 視窗內的對話 + 本輪被 @ 或檢索拉進的檔案／符號 | **短期** = 這串對話 + 本輪從專案拉進的 context，有 token 上限。 |

也就是說：**你眼前的那一長串對話 + 本輪送進模型的內容，就是短期記憶池**；關閉或換新對話就相當於「短期清空」，除非產品另有長期記憶並在下次注入。

### 3.2 長期記憶池的來源與注入方式

| 產品 | 長期記憶池 | 如何注入短期（本輪 context） |
|------|------------|------------------------------|
| **ChatGPT** | **Memory**：用戶顯式要求記住的事 + 可選「參考過往對話」（付費版可參考全部歷史） | 新對話時，已儲存記憶與／或過往對話摘要／檢索結果被注入 system 或 context，模型「記得」偏好與前情。 |
| **Claude** | 有記憶／偏好儲存時：跨 session 的事實與偏好 | 新對話時將相關記憶注入，等同「破碎式長期 + 短期」。 |
| **Gemini** | **Past Chats**：過往對話可被參考以學習偏好與脈絡 | 個人化回應時從過往對話取脈絡，注入當前請求的 context。 |
| **Grok** | **Persistent Memory**（約 2025）：用戶偏好、目標、溝通風格等以 **vector 儲存**，新對話時**語意檢索**相關片段注入 | 新 chat 時從長期用檢索拉出相關記憶注入 prompt；session 內仍為短期。 |
| **Cursor** | **全專案 = 長期記憶池**：Codebase Indexing（檔案與符號經 **embedding** 做語意索引，存於向量庫） | 用戶輸入或 Agent 觸發時，依**當前問題／關鍵字**對專案做 **semantic search**（或 @ 指定檔案／符號），將檢索到的程式碼區塊注入本輪 context；**不**把「對話歷史」當長期，長期是**程式庫本身**。 |

**對照**：Web chat 的長期多是「與用戶有關的記憶（偏好、過往對話）」；Cursor 的長期是**專案程式碼**，對話串只是短期，每輪從專案拉需要的部分進短期。

### 3.3 機制流程簡化

**Web Chat（ChatGPT／Claude／Gemini／Grok）**

```
開新對話 → 從長期記憶池（已存記憶／過往對話）檢索或摘要 → 注入本輪 context（短期）
    ↓
用戶發言 → 短期 = 既有 context + 本輪對話累積（有 token 上限，滿了截斷或壓縮）
    ↓
模型回覆 → 可選：本輪或 session 結束時將「要記住的」寫回長期記憶池
    ↓
關閉對話 → 短期清空；下次新對話再從長期檢索注入
```

**Cursor**

```
開啟專案 → 背景建立 Codebase Index（全專案 = 長期記憶池，embedding + 向量庫）
    ↓
用戶在 Chat 發問或 @file / @code → 以當前問題或 @ 目標對「專案」做語意／路徑檢索 → 檢索結果 + 當前對話串 = 本輪短期記憶池（送進模型）
    ↓
模型回覆（寫程式、解說）→ 僅改動專案檔案；**對話本身不寫入長期**，長期仍是專案索引
    ↓
換新 Chat 或關閉 → 短期（對話串）清空；長期（專案）不變，下次仍可從專案檢索
```

**要點**：Web chat 的長期 = **關於用戶的記憶**；Cursor 的長期 = **專案程式碼**，短期 = 當前對話 + 本輪從專案拉進的片段。兩者都符合「長期記憶池 → 依線索提取 → 注入短期 → 本輪答覆」；差異在「長期存的是什麼」與「誰觸發檢索（系統自動 vs 用戶 @）」。

### 3.4 與本專案對話記憶設計的對照

| 維度 | Web Chat | Cursor | 本專案對話記憶 |
|------|----------|--------|-----------------------------|
| **短期記憶池** | 當前對話視窗 | 當前 Chat + 本輪拉進的程式碼 | 本輪背版 + 檢索記憶 + 最近 N 輪 |
| **長期記憶池** | 用戶記憶／過往對話 | **全專案**（codebase index） | 該主體的 **archival** + **背版 blocks** |
| **注入時機** | 新對話開始時（+ 可選每輪刷新） | 每輪依問題或 @ 從專案檢索 | 每輪回覆前：背版 + 語意檢索 archival |
| **寫入長期** | 顯式記憶 + 可選從對話自動抽取 | **不寫**（對話不反寫專案）；專案由編輯改動 | 對話結束 consolidation → archival + 更新背版 |

本專案較接近 Web chat（長期 = 關於該主體／該使用者的記憶），**subject = 對話主體 id**，長期池不跨 subject。

## 四、Letta 式記憶架構（參考標的）

Letta 以「有狀態、可累積記憶的 Agent」為目標，單一 Agent 可同時進行多場對話，並**跨對話共享記憶**。

### 3.1 兩層記憶

| 層級 | 名稱 | 用途 | 實作要點 |
|------|------|------|----------|
| **核心記憶** | Memory blocks | 身份、偏好、當前狀態、近期摘要；常駐或易塞進 context | 命名區塊（如 identity、summary、policies、history）；Agent 可**自改**（self-editing）；有 `/remember` 等顯式更新與背景 reflection 整理 |
| **長期記憶** | Archival memory | 事實、知識、歷史、跨 session 的對話精華 | **語意檢索**（向量庫）；不常駐 context，需要時用 tool 查詢；Agent 工具有 `archival_memory_search`、`archival_memory_insert` |

### 3.2 與對話的關係

- **Conversations API**：一個 Agent 對應多個 conversation thread；每個 thread 是獨立訊息串，但**記憶跨 thread 共享**。
- **寫入**：對話過程中 Agent 可主動 `archival_memory_insert`；或由背景流程將對話精華寫入 archival。
- **讀取**：組 prompt 時帶入 blocks；需要時用 `archival_memory_search` 拉出相關段落再注入 context。
- **Archival 特性**（官方）：tag 分類、語意搜尋（依意思非關鍵字）、容量可視為不限、Agent 不易刪改（開發者可經 SDK 管理）。

### 3.3 相關連結

- [Letta Memory 文件](https://docs.letta.com/letta-code/memory/)
- [Archival memory 說明](https://docs.letta.com/guides/core-concepts/memory/archival-memory)
- [Conversations: Shared Agent Memory](https://www.letta.com/blog/conversations)
- [Letta AI Memory SDK（GitHub）](https://github.com/letta-ai/ai-memory-sdk)：Python/TypeScript，Subject = 記憶主體，Blocks = 區塊，Messages 非同步更新記憶。
- [Benchmarking AI Agent Memory: Is a Filesystem All You Need?](https://www.letta.com/blog/benchmarking-ai-agent-memory)：向量庫與簡化儲存之比較。

## 五、自建可行性與組件

結論：**可自建**，且所需組件皆為常見開源或雲端服務。

### 4.1 核心組件

| 組件 | 做法 | 備註 |
|------|------|------|
| **Embedding** | 本地：sentence-transformers、BGE、nomic-embed；雲端：OpenAI / 其他 API | 決定 archival 語意檢索品質與成本 |
| **向量儲存** | Qdrant / Chroma / pgvector / Lance / Milvus；或小規模時 SQLite + 簡單向量索引、甚至檔案 | Letta 有文探討「filesystem 是否就夠」 |
| **Blocks 儲存** | 關聯式表或 JSON（如 subject_id, block_name, content, updated_at） | 讀取後在呼叫 LLM 前拼進 system 或前幾則 message |
| **Agent 工具** | `memory_search(query)`、`memory_insert(content, tags)`，可選 `memory_update(block_name, content)` | 由 LLM 在對話中決定何時寫入、查詢 |
| **整理／壓縮（可選）** | 定時或觸發時將「最近 N 輪對話」交給 LLM 產出摘要或更新 blocks | 控制 token、延長有效記憶長度 |

### 4.2 抽象介面建議

- **Subject**：以某個 id（如 user_id、npc_entity_id）為記憶主體，其下掛 blocks 與 archival。
- **讀取**：`GetBackstory(subject_id) string`（組 blocks）；`SearchMemory(subject_id, query string, k int) []string`（archival top-k）。
- **寫入**：`AppendMemory(subject_id, content string, tags []string)`；`UpdateBlock(subject_id, block_name, content string)`（可選）。

### 4.3 難度粗估

| 目標 | 難度 | 說明 |
|------|------|------|
| 僅 archival：插入 + 語意搜尋，作為 Agent 的兩個 tool | 低～中 | Embedding + 向量庫 + 兩個 tool 即可 |
| 加上數個 named blocks，Agent 可讀寫 | 低～中 | 儲存 + 組 prompt 時帶入 + update tool |
| 多 conversation 共用同一 subject 記憶 | 低 | 以 subject_id 分區即可 |
| 背景 reflection／自動壓縮寫回 blocks | 中 | 觸發時機、prompt、寫入策略需設計 |
| 完全對齊 Letta「subconscious agent」非同步更新 | 中～高 | 排程／queue、可能多 worker |

## 七、對話記憶運作流程（本專案目標）

將「對話記憶系統」整合進對話回覆流程：**每次對話都為該主體累積記憶點，背版隨對話越長越生動、有延續性。**

### 7.1 運作流程：人類記憶類比與討論

以下為**人類記憶機制的類比**與據此推得的**對話記憶運作流程**（含提案與補充），寫入文檔供實作對齊。

---

**人類側（類比）**

- **初次**：知覺 → 短期記憶 → 長期記憶 → 歸檔。
- **再臨**：破碎式長期記憶（依線索提取）+ 短期記憶 → 更詳細的長期記憶 → 歸檔。

也就是：當下經驗先進短期，經篩選與鞏固後寫入長期；再次接觸時，用線索從長期拉出「碎片」與當下短期一起作用，形成更豐富的長期表徵並再次歸檔。

---

**提案流程（使用者與對話主體）**

1. **偵測到對話**（使用者發送訊息）→ **接入對話**。
2. **提取背版 + 長期記憶池關鍵字** → **進入短期記憶池**。
3. **收到輸入** → **開始答覆**。
4. **提取關鍵字放入短期記憶池**（一輪對話提取一次？或累計 10K／N token 提取一次？）
5. **偵測對話結束** → **提取短期記憶池關鍵字** → **放入長期記憶池、更新背版**。

---

**術語對照與補充**

| 提案用語 | 對應實作 | 補充 |
|----------|----------|------|
| **長期記憶池** | 該主體的 **archival**（語意檢索）+ **背版 blocks**（identity／summary／relationship） | 持久、依 subject_id 分區；檢索時用「線索」拉出相關片段。 |
| **短期記憶池** | **雙重角色**：(1) **回覆當下的 context**：背版 + 檢索結果（top-k）；**本專案不注入「最近 N 輪對話原文」**，只注入「抽出來的要旨」（你記得朋友*說了什麼*，不是每一句話），輕量且省延遲；(2) **本場「待整理緩存」**：本場對話中要留給結束時 consolidation 的內容（每輪 deconstruct 抽要旨寫入 session／archival）。 | (1) 有 token 上限；用要旨而非原文，對應人類「記得說了什麼」；(2) 不寫入長期，僅在對話結束時一次性整併。 |
| **關鍵字** | 不必限於字面關鍵字；實作上可用**語意檢索**（embedding query）從長期記憶池拉出「與當前話題／使用者本句意思相近」的條目，效果等同「依線索提取的破碎長期記憶」。 | 背版為固定帶入；長期（archival）用使用者輸入或本輪話題做 query，取 top-k 注入短期（context）。 |
| **提取（長期→短期）** | 每次要生成回覆前：背版 + 以當前輸入或話題對 archival 做**語意檢索**，結果與最近 N 輪一併組成「短期記憶池」供本輪使用。 | 可**每輪都重新檢索**（query = 使用者本句），或每 N 輪刷新一次以省成本；若對話偏題不重，每輪刷新較能對準當下話題。 |
| **提取（本輪→短期緩存）** | 本輪使用者輸入與回覆中「值得留給結束時整併」的內容，可**每輪輕量抽取**（0～1 條要旨／關鍵字）寫入本場緩存，或**只累積原始輪次**，結束時再一次性由 LLM／規則產出要旨。 | **一輪一次**輕量抽取：適合控制本場寫入量、避免轟炸場爆量。**累計 N 輪或 N token 再提取**：適合長場，先累積 raw，每 N 輪或每 10K token 做一次小結寫入緩存，結束時再對緩存做總 consolidation。建議：一般場「結束時一次 consolidation」即可；超長場可加「每 N 輪或每 N token 小結一次」。 |
| **對話結束** | **偵測對話結束**（session 結束、關閉對話）→ 從**短期緩存**（本場累積的輪次或已抽要旨）**提取／整併** → 產出 1～3 條要旨寫入**長期記憶池（archival）**，並可**更新背版**（如 relationship block）。 | 對應人類「鞏固」：短期內容經篩選與摘要後歸檔，背版更新為更豐富的摘要，下次再臨時可被「破碎式提取」出來。 |

---

**流程圖（對齊提案）**

```
偵測對話（使用者發送）→ 接入對話
    ↓
提取背版 + 以當前話題／使用者句對長期記憶池（archival）語意檢索 → 結果 + 背版（本專案不帶入最近 N 輪原文，只帶檢索到的要旨）= 短期記憶池（本輪 context）
    ↓
收到使用者本句 → 用短期記憶池生成答覆 → 可選：本輪輕量抽取 0～1 條要旨加入「本場待整理緩存」
    ↓
（重複「收到輸入→答覆→可選抽取」；若設每 N 輪／每 N token 小結，則週期性從緩存產出小結）
    ↓
偵測對話結束
    ↓
從本場短期緩存（或整場輪次）提取／整併 → 產出 1～3 條要旨寫入長期記憶池（archival）→ 可選更新背版（如 relationship）
```

**小結**：類比（知覺→短期→長期→歸檔；再臨→破碎長期+短期→更詳細長期→歸檔）與提案流程一致；「關鍵字」可擴充為語意檢索，「短期記憶池」具雙重角色（本輪 context vs 本場待整理緩存），提取頻率可採「每輪刷新檢索 + 結束時一次 consolidation」或長場加「每 N 輪／N token 小結」。

### 7.2 對照關係

| Letta 概念 | 本專案對應 |
|------------|------------|
| Subject | 一個對話主體（subject_id） |
| Memory blocks | 該主體的**背版**：identity、summary、relationship |
| Archival memory | 該主體的**長期記憶**：具體事件、說過的話、對使用者的印象 |
| 每次對話 | 讀背版 + 檢索記憶 → 回覆；回合結束後寫入本輪精華 |

### 7.3 資料流（簡要）

```
對話請求 → 讀取該主體背版 → 語意檢索該主體記憶（top-k）→ 組 prompt → 回覆
→ 本輪結束後寫入 1～3 條記憶點 → 背版可選更新（summary／relationship）
```

### 7.4 情境：多輪對話—記憶系統運作流程

以下說明**何時讀取記憶、何時寫入**。

**情境**  
使用者與同一對話主體進行多輪對話，例如：

- 第一輪：你好（試探）
- 第二輪：你是誰（問身份）
- 第三輪：我在找某某（正經問答）

接著結束對話（session 結束或關閉）。

---

| 時點 | 系統行為 | 記憶的讀／寫 |
|------|----------|--------------|
| **使用者發送、發起對話** | 接入模型回覆流程。 | **讀**：以該主體的 `subject_id` 讀取**背版**（identity 必備；若有 summary／relationship 一併帶入）。**讀**：用當前使用者輸入或首句當 query，對該主體的 **archival** 做語意檢索，取 top-k（如 3～5 條）注入 prompt。首次對話時 archival 可能為空。 |
| **第一輪** | 組 prompt（背版 + 檢索記憶 + 使用者本句）→ 回覆。 | **寫**（可選）：本輪後可寫入該主體 archival 一條；或暫不寫，留待對話結束一併整理。 |
| **第二輪** | 同上；回覆需用到 identity。 | **寫**（可選）：如「使用者問過身份」。 |
| **第三輪** | 回覆結合背版／記憶。 | **寫**：建議寫入 archival，例如「使用者曾詢問某某」；可帶 tag 方便日後語意檢索。可選：更新 relationship 摘要。 |
| **結束對話** | Session 結束或關閉。 | **寫**（建議）：**對話結束觸發 consolidation**。將本場數輪壓成 1～3 條記憶點寫入該主體的 archival；可選以 LLM 或規則產出關係摘要更新 relationship block。 |
| **下次同一主體再次對話** | 同一 `subject_id` 再次被使用。 | **讀**：仍以同一 subject_id 讀背版 + 語意檢索 archival；可檢索到前次寫入，回覆即可延續前次情境。 |

**要點整理**

- **Subject 不變**：記憶一律綁在 **subject_id**。
- **讀取時機**：每次**要回覆當下**讀背版 + 依當前輸入或話題做 archival 檢索。
- **寫入時機**：可**每輪後**寫 0～1 條，或**對話結束時**一次 consolidation 寫 1～3 條。

---

#### 多輪與對話轟炸

**情境**  
同一場對話延長：前幾輪為有明確意圖的問答，其後**大量連續發問**（對話轟炸），直至結束對話。

**問題**：輪數多（例如 20+ 輪）、若每輪都寫一條會造成 archival 膨脹與噪音；且**回覆時 context 視窗有限**，模型只帶最近 1～2 輪。記憶系統的目標是：**把「值得延續的」壓成少量要旨**，下次對話時用檢索帶出，而非逐句存檔。

**策略：節流、合併、整場 consolidation**

| 原則 | 做法 |
|------|------|
| **不逐句寫** | 不要每輪都往 archival 插一條；否則轟炸 20 輪就 20 條，檢索時容易被無意義內容稀釋。 |
| **區分「有意圖」與「轟炸」** | 前幾輪**有明確意圖**的可各寫 **0～1 條**，或先不寫、留給結束時一併判斷。**連續多輪**明顯試探／重複／轟炸時，**不再逐輪寫**，改為在對話結束時用一條摘要涵蓋。 |
| **寫入節流（可選）** | 同一場對話中，對同一 subject 的寫入可設上限，例如「本場最多寫 M 條」（如 M=5）；超過則只做記憶緩存，結束時再合併成一條 session 摘要。 |
| **對話結束時必做 consolidation** | 本場結束時，將**整場對話**（或最近 N 輪 + 前段摘要）交給 LLM 或規則，產出 **1～3 條**要旨寫入該主體的 archival。這樣下次檢索仍能命中重點，且不會被大量轟炸句佔滿。 |
| **relationship block 更新** | 對該使用者的 relationship 可更新為一句摘要。 |

**要點**：多輪／轟炸情境下，**寫入要節流、要合併**；**結束時整場 consolidation** 產出少量要旨，既保留可延續的資訊，又避免 archival 被單一場轟炸塞爆。

## 八、彙整表：何時用什麼

| 需求 | 建議 |
|------|------|
| 只要「長期事實可語意查」 | 實作 archival（embedding + 向量庫）+ search／insert tool 即可 |
| 要「固定幾塊可編輯的摘要」 | 加 blocks 儲存，組 prompt 時帶入；可選由 LLM 或規則更新 |
| 要「多輪對話共享同一主體記憶」 | 以 subject_id（或 user_id）分區 blocks 與 archival |
| 要「壓縮、不爆 context」 | 定時或觸發：最近對話 → LLM 摘要 → 寫回 blocks 或 archival |
| 不想用向量庫 | 參考 Always On Memory Agent：LLM 整理 + 結構化儲存（如 SQLite） |
| 對話主體越聊越有延續性 | 每主體一 subject；背版 = blocks；對話精華寫 archival；回覆前檢索 + 帶入 prompt；流程見 §7.1，情境見 §7.4，對話記憶與背版—設計 |
