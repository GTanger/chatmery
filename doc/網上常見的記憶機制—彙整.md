# 網上常見的 AI／LLM 記憶機制 — 彙整

> 根據近期網上文章與框架整理，供對照本專案（Chatmery）的設計取捨。  
> 本專案不採用「MD 對話紀錄 + 硬式檢索」或「整段對話歷史注入」，改為「抽出的要旨 → 少數條目注入」，以減輕使用者端等待；以下彙整可幫助判斷未來是否要引入其他機制。

---

## 一、記憶的類型（常見分類）

### 1.1 依「時間／範圍」分

| 類型 | 常見定義 | 典型做法 |
|------|----------|----------|
| **短期記憶（STM）** | 當前 session 的對話歷史 | 保留最近 N 輪訊息、或 session 內 buffer；session 結束即丟棄。 |
| **長期記憶（LTM）** | 跨 session 持久、可累積的脈絡 | 存進資料庫（向量庫、圖庫、關聯式）；需要時再檢索注入。 |
| **Context 視窗** | 模型「當下看得見」的內容 | 等同工作記憶／RAM；有限、貴，不宜塞滿整段對話。 |

不少文章會強調：**不要把「短期」等同於「把最近 N 句原文塞進 context」**——那樣會爆 token、延遲高；更常見的是用「摘要」或「抽出的要旨」代表「剛才說了什麼」。

### 1.2 依「認知／功能」分（Mem0、多篇架構文）

| 類型 | 內容 | 實作難度 |
|------|------|----------|
| **Episodic（情境記憶）** | 具體事件、對話輪次、決策、tool 呼叫 | 較易：存＋依時間或相關性檢索。 |
| **Semantic（語意記憶）** | 從經驗「蒸餾」的知識：使用者偏好、已棄用 API 版本等 | 較難：要從 raw 事件產出摘要／事實。 |
| **Procedural（程序記憶）** | 怎麼做某件事、有效步驟（例如一連串 API 呼叫） | 需結構化或範例。 |
| **Short-term** | 當前 session 對話；通常 transient | 可為 buffer 或摘要。 |
| **Long-term** | 跨 session 持久 | 需儲存與檢索架構。 |

## 二、常見架構與實作方式

### 2.1 向量庫（Vector Store）— 最常見

- **做法**：把文字過 embedding 模型 → 存成向量；查詢時用當前 query 的向量做相似度搜尋（如 cosine），取 top-k 條注入 context。
- **優點**：語意相近即可召回，不必完全關鍵字匹配；實作成熟（Pinecone、Weaviate、Qdrant、pgvector 等）。
- **缺點**：不擅長「有向關係」（例如 A 是 B 的 capital）；若存的是整段對話或大 chunk，檢索與注入仍可能變重、使用者體感「等了又等」。

### 2.2 GraphRAG / 圖結構記憶

- **做法**：用圖資料庫（如 Neo4j）存節點與邊，例如 `(Paris) --[CAPITAL_OF]--> (France)`；檢索時走圖的關係做多跳推理。
- **適用**：需要明確關係、多跳問答（例如供應鏈、知識圖譜）。
- **與向量**：常與向量併用——向量負責「模糊相關」，圖負責「精確關係」。

### 2.3 混合架構（L1 / L2 / L3 等分層）

部分文章提到生產環境會分層，例如：

- **L1（Hot）**：當前任務狀態，極低延遲（如 Redis）。
- **L2**：語意檢索（向量庫），約 20ms 級。
- **L3**：長期事件／episodic 日誌（如 Pinecone serverless）。

同一套 embedding 貫穿，差在儲存與延遲需求。

### 2.4 Reflection / 反思機制（Stanford Generative Agents 等）

- **概念**：不只存「觀察」（raw 事件），還週期性用 LLM 產出**高層次反思**（例如「使用者常在一點吃午餐」），再存進記憶。
- **Memory stream**：觀察 + 反思 + 計畫，每筆帶時間戳；檢索時可依重要性（importance）、新近度（recency）、與當下相關性（relevance）排序。
- **效果**：讓 agent 行為更連貫、像「有在思考」，而不是單純 regurgitate 資料。

### 2.5 「作業系統」類比：LLM 自管記憶

- **概念**：把 LLM 當 OS；context 視窗 = RAM（貴、有限），外部儲存 = Disk（便宜、大）。
- **做法**：透過 function call 讓模型自己決定「什麼要寫入長期」、「何時查歷史」，避免 context 被雜訊塞滿，同時能存取大量過去資料。

### 2.6 摘要／壓縮（Summary / Compression）

- **做法**：對話或事件超過一定 token 就觸發摘要（例如 LangChain 的 `ConversationSummaryBufferMemory`、`SummarizationNode`），用摘要取代舊訊息，只保留最近幾輪原文。
- **參數常包括**：`max_tokens_before_summary`、`max_summary_tokens`。
- **目的**：控制 context 長度、兼顧「記得大概說了什麼」與延遲／成本。

## 三、框架與產品（常見提及）

| 名稱 | 記憶相關要點 |
|------|----------------|
| **LangChain** | `ConversationBufferMemory`、`ConversationSummaryBufferMemory`、`VectorStoreRetrieverMemory` 等；可搭配 checkpointer 做 thread-scoped 短期記憶。 |
| **LangGraph** | Short-term = thread 內訊息歷史（persist 成 checkpoint）；Long-term = 跨 session 的 store。可設「超過 token 就 summarization」。 |
| **Mem0** | 主打「Memory as a Service」：向量 + 可選 Graph Memory；自動更新／覆寫衝突記憶、衰減不重要記憶；與 LangChain、OpenAI 等整合。 |
| **LlamaIndex** | 索引策略可組合：list index on top of vector store on top of graph。 |
| **OpenAI ChatGPT Memory** | 使用者「請記住」的顯式記憶 + 自動偵測重要資訊；付費版可參考過往對話做個人化；可開「Temporary Chat」不寫入記憶。 |
| **Generative Agents (Stanford)** | Memory stream（觀察 + 反思 + 計畫）+ 檢索時用 importance / recency / relevance 排序；示範在 Smallville 沙盒。 |
| **AutoGPT** | 早期示範：agent 把想法寫進檔案／DB「睡覺」，下一輪再讀，強調記憶驅動的迴圈。 |

## 四、常見結論與取捨

1. **RAG 不等於記憶**：RAG 解決「從文件裡找答案」；記憶解決「跨輪、跨 session 的脈絡與個人化」。兩者常一起用，但記憶需要「寫入、更新、衰減、檢索」一整套。
2. **Context 視窗是稀缺資源**：塞整段對話或大塊 MD 會貴、慢、且「Lost in the Middle」效應；多數做法是「精選少量條目」或「摘要」再注入。
3. **「短期」不必是原文**：短期記憶可以是「本場抽出的要旨」或「最近幾輪的摘要」，而不是把最近 N 句一字不差送進 API——這樣才容易在即時對話管道保持輕量。
4. **長期記憶需要結構**：純 key-value 不夠；向量、圖、或混合，依產品需求選擇。本專案目前用 JSONL + 關鍵字檢索，屬於輕量、可日後換成向量或圖。

## 五、與本專案（Chatmery）的對照

| 項目 | 網上常見做法 | 本專案選擇 |
|------|--------------|------------|
| 短期記憶 | 常為「最近 N 輪原文」或「buffer + 摘要」 | **不傳原文**；每輪 deconstruct 抽要旨寫 session／archival，回覆時只注入檢索到的幾條。 |
| 記憶來源 | 有的依賴整份 MD 對話紀錄 + 直接檢索 | **不以 MD 對話紀錄為回覆時記憶**；避免「等了又等」。 |
| 長期儲存 | 向量庫、圖庫、episodic 日誌 | 目前 JSONL + 關鍵字檢索；預留可換成向量／圖。 |
| 背版 | Memory blocks（identity / summary / relationship） | SOUL.md、MEMORY.md + archival 檢索結果。 |
| 目標場景 | 各式 agent、客服、教育、CRM | 約 4B 小模型 + Web；輕量、少等。 |

之後若實測仍覺得「等了又等」，可再從本彙整裡挑機制微調（例如：加摘要層、改檢索條數、或換成向量檢索但嚴格限制條數與長度）。
